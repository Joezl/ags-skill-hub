ArcGIS Skill Install — Auth Problem & Solution

Why the current OAuth flow breaks inside AI agents, and how the Token Relay fixes it

The Problem: Agent can't catch the OAuth redirect Current flow spins up a tiny web server on port 8976 waiting for ArcGIS to redirect back after login. AI agents (Claude Code, Codex) run in sandboxes — they can't open browsers or run localhost servers that the user's browser can actually reach.
The Fix: Skill Hub as the OAuth broker (Token Relay) The Skill Hub — which already knows how to do ArcGIS OAuth — handles the browser part. The agent just polls the Skill Hub waiting for "did the user log in yet?" The token never needs to cross the browser-to-agent gap directly.

❌ Current (Localhost Callback)

😰Agent starts a tiny web server on port 8976 inside its sandbox
🌐Opens ArcGIS OAuth — user logs in
↩️ArcGIS redirects browser to localhost:8976
💥Browser hits localhost:8976 on the user's machine, not the agent's sandbox — connection refused
🚫Login stuck. No token received. Install fails.

✅ Proposed (Token Relay)

📋User pastes install block → agent calls Skill Hub to create a relay session
🔗Agent shows user one link: skillhub/auth/relay?session=xyz
🌐User clicks link, logs in via normal ArcGIS OAuth — all in browser
🔐Skill Hub stores the token keyed to session ID (short-lived, single-use)
✅Agent polls Skill Hub until token is ready → downloads zip → installs skill

What the user experience looks like end-to-end

1. User visits Skill Hub, finds a skill, clicks "Copy Install Prompt"
2. User pastes into Claude Code or Codex — agent sees the ARC_SKILL_INSTALL block
3. Agent: "This skill requires ArcGIS login. Open this link in your browser: https://skillhub/auth/relay?session=abc123"
4. User clicks link, ArcGIS OAuth opens, logs in — page says "Done! Return to your agent."
5. Agent was polling in the background — token arrives automatically. Downloads and installs skill. Done.
